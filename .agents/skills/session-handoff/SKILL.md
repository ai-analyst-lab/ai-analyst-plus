# Session Handoff

## Purpose

Create a compact but complete “save game” for long Codex workflows. The handoff lets a later
session, `$resume-pipeline`, or another agent recover the task, local files, external resource
IDs, blockers, and next steps.

This is the Codex-native counterpart to the legacy Claude session-handoff skill.

## When to use

Use proactively:

- after creating any external resource, especially Google Docs, Slides, Drive files, or
  Notion pages;
- after completing major pipeline phases;
- before long-running or risky operations;
- when a workflow has 15+ tool calls or is likely to hit context limits;
- when the user says “save state,” “pause,” “come back later,” “bookmark this,” or similar;
- when handing off between analysis, deck, export, or integration phases.

## Output path

Always write the complete snapshot to exactly:

```text
working/session_state.yaml
```

Do not use alternate filenames. Overwrite the file with a fresh complete snapshot each time.

## Required content

Always include:

- dataset name and non-sensitive connection summary;
- task description;
- pipeline/progress status;
- all external resource IDs and URLs;
- local file inventory for key `working/` and `outputs/` files;
- auth state and account/email when known;
- known issues/blockers, or `None`;
- step-by-step resume instructions.

Do not include secrets, passwords, API tokens, private keys, or raw credentials.

## Template

Use this structure every time:

```yaml
# Session State — auto-generated by session-handoff skill
# Last updated: YYYY-MM-DD HH:MM TZ

dataset: "<active dataset or unknown>"
task: "<brief description of current work>"

resources:
  google_doc:
    id: "<id or null>"
    title: "<title>"
    url: "<url>"
    status: "<plain-English status>"
  google_slides:
    id: "<id or null>"
    title: "<title>"
    url: "<url>"
    status: "<plain-English status>"
  notion_page:
    id: "<id or null>"
    title: "<title>"
    url: "<url>"
    status: "<plain-English status>"
  drive_files:
    - id: "<id>"
      name: "<filename>"
      url: "<url if known>"

pipeline:
  run_id: "<run id if known>"
  run_dir: "<working/runs/... if known>"
  last_completed_step: "<phase or agent>"
  current_step_status: "<partial progress>"
  next_step: "<next action>"
  steps_remaining:
    - "<step>"

local_files:
  outputs:
    - "outputs/<file>"
  working:
    - "working/<file>"
  charts:
    - "outputs/charts/<file>"
  data:
    - "outputs/data/<file>"

auth:
  google_email: "unknown — verify at resume"
  notion_workspace: "unknown — verify at resume"
  last_verified: "YYYY-MM-DD or unknown"
  notes: "<auth caveats>"

issues:
  - "None — all work completed successfully"

resume:
  command: "$resume-pipeline"
  steps:
    - "Read working/session_state.yaml"
    - "Verify auth and external resources"
    - "Continue from: <pipeline.next_step>"
```

If a resource type is irrelevant, include it with null/unknown fields or omit it only when
clearly unnecessary. Include every external ID immediately after creation.

## How to resume from handoff

At a later session start:

1. Read `working/session_state.yaml` if it exists.
2. Verify local files still exist.
3. Verify external resources still exist when tools/auth are available.
4. Verify auth/account if using Google or Notion.
5. Report a concise status summary:

```text
Resuming from previous session:
- Task: ...
- Dataset: ...
- Last completed: ...
- External resources: ...
- Next step: ...
- Auth: OK / needs verification
```

Then continue with `$resume-pipeline`, `$export`, or the named next skill.

## Rules

- Write state immediately after every external resource creation.
- Include URLs as well as IDs.
- Describe status in plain English.
- Keep the state compact; reference files rather than embedding full narratives.
- Overwrite, do not append.
- Be comprehensive even for simple handoffs.
- Redact secret-looking values.
